VST-IDE

The extensible, semi-automated C verification toolchain. A C program annotated with separation-logic assertions is symbolically executed. When there are problems cannot be solved by built-in execution—say reading and writing memory or computing frames—it fires a contextual rewriting (specified by the developer using a pattern-based DSL) to solve separation-logic entailments. Finally, it exports remaining (not solved by rewriting and SMT solver) verification conditions to Coq. The toolchain has been used in real-world operating system verification and eBPF verification (NSDI submission).

I am one of the core contributors of VST-IDE. I coded the C frontend, the assertion-language elaborator, the type system of the assertion language, and a compiler targeting eBPF bytecode. The compiler also compiles C annotations to eBPF ones.

Scope

This is the outcome of my independent research in Peking University.

Justin Pombrio, Shriram Krishnamurthi, Mitchell Wand (1) used preorders to model scopes, (2) designed a binding language to specify scoping rules base on this idea, and (3) showed how to infer scoping rules of new constructs introduced by syntactic sugars. However, their claim on the uniqueness of derivations is wrong. As consequences, their inference algorithm is unsound. We can restrict their binding language to recover uniqueness, but then it is not strong enough to guarantee transitivity require in preorders. Also, some scoping rules cannot be expressed even the original language.

We argue that sticking to preorders prohibit some interesting scopes. Further, we propose a simple but powerful generalization to their binding language, effectively using multiple ports instead of one. It enables some scoping previous not expressible, including match and do. As suggested in the paper, it also includes support for list patterns; but it is only useful in our generalized setting. We also extend the inference algorithm (now sound) accordingly and prove a complexity result.

We observe that a scope is expressible in our binding language precisely when it is a regular language in some sense. It is in contrast to the original one, where one can create nonlocal behaviour and break hygiene. It also mitigates the additional complexity of our language extension. Some sanity checks and optimizations to scope checking are clearer through the lens of formal language theory.

We implemented the system described in the report. We also implement the original version in Osazone with integration of Emacs: users can actually find-definition-at-point in Emacs if she crafted a Tree-sitter parser.

CS1959-compiler

Native compiler of a subset of Scheme. Based on course P523 in Indiana University, but with more optimizations:

  • Constant propagation, constant folding, and dead code elimination;

  • Better register allocation through iterated register coalescing;

  • Merge adjacent allocations;

  • Closure optimizations;

  • Handle (nested) letrec more reasonably;

and more features:

  • Division, characters, and symbols;

  • Undelimited continuations by capturing the whole stack;

  • Stop-the-world copying garbage collection;

  • Encode large literals to binary and decode them at run-time;

  • A JIT variant which emits bytecode, which is then interpreted and/or dynamically compiled.

I know this project from Yin Wang and ended up beating him (at least in generated code size; definitely not in programming elegancy). On some test cases (e.g. the 3-story tower of interpreters), performance is a little better than unsafe Chez (not so fair because it has much more considerations).

As a TA, I adopted this project to replace the old compiler project. Three of my classmates and I took the old one. It was not a good experience: there were almost no guidance and the source language was simple.

LTL

Turing

A DSL for Turing machines. Each rule is in the form

((state symbol1 ... symboln) (state' symbol2' ... symboln' move1 ... moven))

if there are n tapes and the first tape is read-only. Store and index transition rules in a trie.
